# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Source of truth

`config.org` is the single source of truth for all system configuration. **Never edit `flake.nix`, `hosts/**/*.nix`, or `modules/*.nix` directly** — those files are generated by tangling from `config.org`.

Edit flow: update `config.org` → run `./scripts/tangle-config.sh` → validate/build.

## Validation commands

```bash
# Regenerate Nix files from config.org
./scripts/tangle-config.sh

# Fast evaluation (check flake structure)
nix --extra-experimental-features 'nix-command flakes' flake show path:$PWD

# Dry-run build (resolve full system without building)
nix --extra-experimental-features 'nix-command flakes' build --dry-run path:$PWD#nixosConfigurations.nixos.config.system.build.toplevel

# Build without activating
nix --extra-experimental-features 'nix-command flakes' build --no-link path:$PWD#nixosConfigurations.nixos.config.system.build.toplevel

# Test-activate (temporary, reverts on reboot)
sudo nixos-rebuild test --flake .#nixos

# Apply permanently
sudo nixos-rebuild switch --flake .#nixos
```

There is no lint script or unit/integration test runner — validation is whole-system. Use `path:$PWD#...` targets (not `.#...`) when untracked files are involved.

## Architecture

- `flake.nix` pins `nixpkgs`, `nixos-wsl`, and `copilot-cli`; exposes a single host at `nixosConfigurations.nixos`.
- `hosts/nixos/default.nix` is the host entrypoint — it assembles the system from focused modules.
- Modules under `modules/` are grouped by user-facing function:
  - `base.nix` — Nix settings, unfree allowance, shell defaults, `system.stateVersion`
  - `wsl.nix` — WSL-specific behavior only
  - `development.nix` — general development tooling
  - `ai-coding.nix` — AI coding CLIs (Copilot CLI via flake input, Claude Code, waza, shared Node.js runtime)
  - `emacs.nix` — Emacs-centric toolchain: native deps for `vterm`, Python, dictionaries, fonts

## Key conventions

- When changing `config.org`, preserve context in prose near the affected chunks (what changed, why, functional outcome).
- Architecture decisions go in `config.org` as dedicated functional sections (Org tags), not standalone notes.
- Use uppercase Org tags only. Tag `:TANGLE:` for pure assembly/tangling module headers.
- Do not use `=` inline markup in Org heading text.
- Split Babel chunks by functionality (small named chunks for inputs, package groups, toggles), assembled via `:tangle` blocks — not extra `*-file` wrapper chunks.
- Post-processing cleanup is handled by `./scripts/tangle-config.sh`; do not hand-format generated `.nix` files.
- Group modules by user-facing function. Add packages to the functional module they support (e.g. AI tools → `ai-coding.nix`, not a new `packages.nix`).
- Prefer packages from pinned `nixpkgs` before adding a new flake input. When an upstream flake already exposes the package, consume it directly from `packages.${pkgs.system}`.
- Keep WSL-specific behavior in `wsl.nix`. Keep AI tooling in `ai-coding.nix`. Keep host composition in the Host section of `config.org`.
- `nixpkgs.config.allowUnfree = true` is already set — use normal package references for unfree packages.
- Preserve `system.stateVersion` unless explicitly migrating a NixOS release.
- Avoid git write actions unless the user explicitly requests them.
- When a request changes how configuration or document structure should be organized, update `.github/copilot-instructions.md` in the same change.
