#+TITLE: NixOS Literate Configuration
#+PROPERTY: header-args:nix :comments no :mkdirp yes :noweb yes

This document is the canonical source for the full NixOS configuration.
In literate style, we first explain each concern, then weave named code chunks
into concrete files via Org Babel tangling.

Functionally, this machine profile is organized around five outcomes:
- a stable NixOS-on-WSL base,
- reproducible host composition,
- core development tooling,
- Copilot CLI availability,
- and an Emacs-first working environment.

* Flake: dependency graph and system entrypoint

The flake has three responsibilities: declare repository intent, pin inputs, and
define outputs for the =nixos= host.

From a functional perspective, this section is the contract that turns this
repository into a reproducible workstation definition: same inputs, same module
graph, same host outcome.

** Acceptance criteria (Gherkin)

These scenarios define what must hold true for flake-level behavior.

#+begin_example
Scenario: Flake exposes the nixos host configuration
  Given the repository is in a tangled state
  When I run "nix --extra-experimental-features 'nix-command flakes' flake show path:$PWD"
  Then the output includes "nixosConfigurations"
  And the output includes "nixos"

Scenario: Flake resolves the system toplevel derivation
  Given the repository is in a tangled state
  When I run "nix --extra-experimental-features 'nix-command flakes' build --dry-run path:$PWD#nixosConfigurations.nixos.config.system.build.toplevel"
  Then the command exits successfully
#+end_example

Verification mapping:
- Scenario "Flake exposes the nixos host configuration" -> `nix ... flake show path:$PWD`
- Scenario "Flake resolves the system toplevel derivation" -> `nix ... build --dry-run path:$PWD#nixosConfigurations.nixos.config.system.build.toplevel`

modules are active for the =nixos= host without embedding option logic here.
