#+TITLE: NixOS Literate Configuration
#+PROPERTY: header-args:nix :comments no :mkdirp yes :noweb yes

This document is the canonical source for the full NixOS configuration.
In literate style, we first explain each concern, then weave named code chunks
into concrete files via Org Babel tangling.

Functionally, this machine profile is organized around five outcomes:
- a stable NixOS-on-WSL base,
- reproducible host composition,
- core development tooling,
- Copilot CLI availability,
- and an Emacs-first working environment.

* Gherkin coverage model

To keep acceptance criteria useful for generated tests, this document uses the
following rule:
- every **behavior-affecting snippet** gets an `Acceptance criteria (Gherkin)`
  subheader near that snippet;
- **structural or tangling-only snippets** do not get their own criteria.

Behavior-affecting snippets covered with Gherkin in this document:
- `flake-output-modules`
- `host-imports`
- `module-base-nix-features`
- `module-base-unfree`
- `module-base-shell`
- `module-base-auto-upgrade`
- `module-base-state-version`
- `module-wsl-enable`

Structural/tangling-only snippets intentionally excluded from direct criteria:
- flake assembly/meta snippets (`flake-description`, `flake-input-*`, `flake-inputs`,

- module argument/assembly snippets (`module-*-args`, `module-*-body`,
  `module-development-packages`, `module-copilot-packages`, `module-emacs-toolchain`)
- final `:tangle` blocks that only weave code into generated files

* Flake: dependency graph and system entrypoint

The flake has three responsibilities: declare repository intent, pin inputs, and
define outputs for the =nixos= host.

From a functional perspective, this section is the contract that turns this
repository into a reproducible workstation definition: same inputs, same module
graph, same host outcome.

Enable unattended system upgrades and prefer rolling package refreshes.

Per request, this machine should update itself automatically and track newer
package revisions over time. To preserve local host behavior while still
getting rolling-style package updates, auto-upgrade rebuilds from this local
flake path and refreshes the =nixpkgs= input during each upgrade run.

#+name: module-base-auto-upgrade
#+begin_src nix :tangle no
system.autoUpgrade = {
  enable = true;
  dates = "daily";
  randomizedDelaySec = "45min";
  flake = "/home/nixos/nixos";
  flags = [
    "--update-input"
    "nixpkgs"
  ];
  allowReboot = false;
};
#+end_src

**** Acceptance criteria (Gherkin) for =module-base-auto-upgrade=

#+begin_example
Scenario: Auto-upgrade is enabled on a daily schedule without forced reboot
  Given the host configuration is evaluable
  When I evaluate auto-upgrade settings
  Then "system.autoUpgrade.enable" equals true
  And "system.autoUpgrade.dates" equals "daily"
  And "system.autoUpgrade.allowReboot" equals false

Scenario: Auto-upgrade refreshes nixpkgs from the local flake path
  Given the host configuration is evaluable
  When I evaluate the auto-upgrade flake source and flags
  Then "system.autoUpgrade.flake" equals "/home/nixos/nixos"
  And "system.autoUpgrade.flags" includes "--update-input"
  And "system.autoUpgrade.flags" includes "nixpkgs"
#+end_example

Verification mapping:
- Scenario "Auto-upgrade is enabled on a daily schedule without forced reboot" -> `nix eval --json path:$PWD#nixosConfigurations.nixos.config.system.autoUpgrade.enable` + `nix eval --raw path:$PWD#nixosConfigurations.nixos.config.system.autoUpgrade.dates` + `nix eval --json path:$PWD#nixosConfigurations.nixos.config.system.autoUpgrade.allowReboot`
